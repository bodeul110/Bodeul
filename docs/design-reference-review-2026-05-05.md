- 보안, 운영, QA 요구사항과 충돌하는 디자인 요소는 그대로 반영하지 않는다.

## 현재 구현과 크게 맞는 축

- 흰 배경과 파란 포인트 색을 중심으로 한 기본 톤
- 큰 인사 헤더와 요약 카드 중심의 홈 화면 구조
- 매니저 내 페이지에서 서류 상태와 계정 정보를 한 화면에 모으는 방향
- 권한 안내, 예약, 진행 상태, 서류 업로드를 독립 카드로 나누는 구성

즉, 디자인 방향이 완전히 다른 상태는 아니고, 현재 구현이 이미 같은 축으로 정리돼 있다.

## 바로 반영 가치가 있는 차이

### 1. 사용자 화면의 개발/연동 표시 축소

- 현재 앱 주요 화면에는 `Firebase 연동 모드` 배지가 상단에 자주 보인다.
- 디자인 레퍼런스에는 이 노출이 없다.
- 내부 테스트에는 유용하지만 사용자 시연 화면에서는 산만하다.

권장:

- debug 전용이거나 내부 운영 계정에서만 보이도록 조건부 노출
- 일반 사용자 데모 화면에서는 숨김

### 2. 권한 안내 화면의 위계 강화

- 레퍼런스는 필수/선택 권한을 카드 단위로 더 분명하게 구분한다.
- 현재 구현은 텍스트 설명 비중이 크고 카드 강조가 약하다.

권장:

- `필수`, `선택` 라벨을 더 직접적으로 표시
- 아이콘, 제목, 한 줄 설명 구조로 간결화
- 하단 안내문은 유지하되 시각적 강조를 줄이기

### 3. 매니저 홈 액션 카드의 시각 밀도 조정

- 레퍼런스는 `서류 등록`, `스케줄 등록`, `과거 경력`, `문의하기` 카드가 더 명확한 아이콘과 간격으로 정리돼 있다.
- 현재 화면도 기능은 있지만 카드 간 차이가 약하고 정보 밀도가 낮다.

권장:

- 카드 아이콘과 상태 칩 대비 강화
- 첫 화면에서 보이는 액션 카드 수와 여백 재조정
- 소개/프로모션 카드와 기능 카드의 구분 강화

### 4. 매니저 서류 업로드 섹션 강조

- 레퍼런스는 `신분증 업로드`, `자격증`, `범죄경력 조회서`를 더 명확한 업로드 흐름으로 보여준다.
- 현재 구현은 기능은 충분하지만 카드 형태와 업로드 상태 전달이 다소 보수적이다.

권장:

- 문서별 상태 칩과 필수 여부 표시 강화
- 업로드 버튼과 파일명/업로드 시각 표시 간 위계 조정
- 심사 상태와 원본 파일 상태를 시각적으로 분리

## 지금은 유지하는 것이 맞는 차이

### 1. 관리자 로그인 숨김 진입

- 디자인상 관리자 진입이 직접 보이더라도, 현재 정책상 일반 사용자 경로에서 관리자 로그인을 노출하지 않는 쪽이 맞다.
- 이 부분은 디자인보다 운영/보안 요구를 우선한다.

### 2. 관리자 웹의 별도 동작 기준

- 현재 ZIP에는 관리자 웹 전용 화면 명세가 충분하지 않다.
- 그래서 웹은 모바일 디자인을 그대로 따라가기보다 현재 운영 화면 요구를 기준으로 유지하는 편이 맞다.

### 3. `Main.png`의 어두운 서류 인증 화면 톤

- 이 화면은 시각적 참고는 가능하지만, 현재 앱 전체 톤과 직접 연결하기에는 기준이 약하다.
- 단일 화면만 어두운 테마로 분기하는 것은 일관성 비용이 더 크다.

## 후순위 polish 후보

- 로그인/역할 선택 화면의 여백과 아이콘 정리
- 카드 그림자, 반경, 내부 패딩 일관성 점검
- 빈 상태, 오류 상태, 업로드 실패 상태 문구와 시각 스타일 통일
- 관리자 웹 목록과 상세 모달의 시각 위계 정리

## 추천 적용 순서

1. `Firebase 연동 모드` 같은 내부용 표시 축소
2. 권한 안내 화면 시각 polish
3. 매니저 서류 업로드/내 페이지 카드 정리
4. 매니저 홈 액션 카드 정리
5. 로그인/역할 선택 polish

## 결론

- 최신 피그마 산출물은 현재 구현과 방향이 크게 어긋나지 않는다.
- 다만 지금 바로 유효한 것은 `시각 polish` 중심의 차이이고, 기능 구조를 다시 바꿔야 할 정도의 충돌은 없다.
- 따라서 이 레퍼런스는 `기능 재설계 기준`이 아니라 `화면 완성도 보강 기준`으로 쓰는 것이 맞다.
