# 디자인 레퍼런스 검토 메모

기준일: 2026-05-22

중요한 전제:

- 이 ZIP은 참고 디자인이다. 확정 명세로 바로 취급하지 않는다.
- 기능 구조와 권한/운영 정책은 `../local/보들_플랫폼_기능설명서.pdf`, [화면 개편 목표 정리](../planning/screen-restructure-target.md), [현재 구현 상태](../status/implementation-status.md)를 우선 기준으로 둔다.
- 디자인이 보안, 운영, QA 요구와 충돌하면 현재 정책을 우선한다.

## 현재 구현과 크게 맞는 방향

- 환자/보호자 홈과 매니저 홈을 각자 독립된 메인 화면으로 다루는 구조
- 카드 단위로 주요 기능을 분리하는 방식
- 권한 안내, 예약, 진행 상태, 서류 업로드를 각각 별도 흐름으로 나누는 방식
- 관리자 기능을 사용자 앱 메인 흐름과 섞지 않는 방향

즉, 최신 ZIP은 현재 구현 방향을 뒤집는 자료가 아니라 `시각 polish` 참고 자료에 가깝다.

## 바로 반영 가치가 있는 차이

### 1. 사용자 홈의 시각 위계

- 최신 보드는 환자/보호자 홈에서 상단 핵심 카드와 하단 기능 카드를 더 강하게 분리한다.
- 현재 구현도 기능은 충분하지만 카드 간 밀도와 강조 차이가 상대적으로 약하다.

권장:

- 상단 핵심 상태 카드와 하단 기능 카드의 여백/배경 대비를 더 벌리기
- 예약 대기, 진행 중, 완료 후 상태에 따라 카드 배지를 더 명확히 분리하기

### 2. 매니저 홈의 카드 정리

- 최신 보드는 `실시간 매칭`, `서류/스케줄`, `경력/문의`의 기능군이 더 뚜렷하다.
- 현재 구현은 기능은 있으나 카드 크기와 정보 밀도에서 조금 더 묶을 수 있다.

권장:

- `실시간 매칭` 관련 카드와 `관리` 카드군을 시각적으로 더 구분하기
- 매니저 홈에서 서류 상태와 일정 상태를 요약 배지로 더 직접 보여주기

### 3. 권한 안내 화면 표현

- 최신 보드의 `권한 설정`은 권한 항목의 분류와 목적 설명이 더 직접적이다.
- 현재 화면은 이미 개선됐지만 `필수/선택`, `왜 필요한지`, `현재 버전 사용 여부`를 한 번 더 단순화할 수 있다.

권장:

- 실제 사용 중인 권한과 후속 확장 권한을 더 명확히 나누기
- 설명 문장 길이를 조금 더 줄이고 카드 제목 중심으로 정리하기

### 4. 사용자 설정/마이페이지 표현

- ZIP 안의 `사용자 설정` 보드는 프로필/기록/정산/설정의 분리감을 더 강하게 준다.
- 현재 앱 마이페이지 계열은 기능은 있으나 시각적으로는 아직 정보 묶음이 평평한 편이다.

권장:

- 프로필, 이용 내역, 정산/후속, 고객지원 묶음을 카드 또는 섹션 단위로 더 분리하기

## 지금 유지하는 게 맞는 차이

### 1. 관리자 로그인 숨김 진입

- 디자인에서 관리자 경로가 보인다고 해도, 현재 정책처럼 일반 사용자 경로에 관리자 로그인을 직접 노출하지 않는 편이 맞다.

### 2. 관리자 웹의 별도 구조

- 최신 ZIP은 주로 사용자 앱 보드 중심이다.
- 관리자 심사/운영은 현재처럼 별도 웹 기준으로 유지하는 쪽이 맞다.

### 3. 실시간 위치/채팅/지도 표현

- 기능설명서 기준으로는 중요하지만 현재 구현 상태는 부분 구현이다.
- 디자인 보드가 있어도 지도/실시간 채팅을 시각만 먼저 확대하는 것보다 실제 연동 범위를 먼저 확정하는 편이 맞다.

## 적용 우선순위

1. 환자/보호자 홈 카드 위계 정리
2. 매니저 홈 카드군 정리
3. 권한 안내 문장/배지 단순화
4. 마이페이지/설정 화면 묶음 정리

## 결론

- `design_refs/보들 가이드.zip`은 현재 구현을 크게 뒤집는 자료가 아니다.
- 최신 기능설명서 기준 구조는 유지하고, 디자인 ZIP은 `화면 polish 기준`으로 사용하는 게 적절하다.
- 다음 디자인 반영은 기능 추가보다 `홈/권한/설정 화면의 시각 위계 정리` 중심이 맞다.
