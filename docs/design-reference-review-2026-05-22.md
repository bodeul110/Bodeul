## 이번 디자인에서 분명해진 축

### 1. 인증/공통 화면

- 스플래시, 로그인, 역할 선택, 권한 안내 모두 `짧은 설명 + 큰 카드 1~2개 + 단일 주 액션` 구조다.
- 현재 앱도 흐름은 맞지만, 카드 간격과 강조 단계가 더 단순해질 필요가 있다.

### 2. 환자/보호자 메인 화면

- 상단에 현재 상태를 강하게 보여주는 핵심 카드가 있고,
- 그 아래에 2x2 퀵 액션 카드,
- 마지막에 소개/보조 콘텐츠 카드가 오는 구조가 반복된다.
- 지금 구현은 기능은 충분하지만, 상단 상태 카드와 하단 기능 카드의 위계가 더 선명해야 한다.

### 3. 환자 상세/진행 화면

- 예약 확정 이후 화면은 `상태 요약 -> 진행 단계 -> 현장 메모/안내 -> 주 액션` 흐름이 강하다.
- 현재 앱은 데이터는 많지만, 한 화면에서 정보 밀도가 조금 높다.
- 특히 진행 상태 카드와 후속 안내 카드의 구획을 더 분리할 필요가 있다.

### 4. 매니저 홈/가이드

- 매니저 홈은 `현재 대기/진행 상태 카드`가 가장 중요하고, 그 아래 액션 카드가 2x2로 따라온다.
- 매니저 가이드는 `왼쪽 단계 레일 + 오른쪽 단일 포커스 카드` 구조가 핵심이다.
- 현재 구현은 이 방향으로 가고 있지만, 카드 묶음과 버튼 우선순위를 더 정리할 수 있다.

### 5. 서류 등록

- 서류 등록은 파일 종류 구분, 현재 상태, 다음 액션이 한 묶음으로 보여야 한다.
- 현재 매니저 서류 등록과 내 페이지는 기능은 충분하지만, 상태 카드와 업로드 CTA 위계를 더 좁혀도 된다.

## 유지해야 하는 차이

### 1. 관리자 숨김 진입

- 디자인에서 관리자 경로가 직접 보이더라도, 현재 정책처럼 일반 사용자 경로에서 관리자 로그인을 숨기는 쪽이 맞다.

### 2. 관리자 웹 분리

- 디자인 세트는 앱 화면 기준이다.
- 관리자 심사/운영은 지금처럼 `admin-web` 별도 기준으로 유지하는 편이 맞다.

### 3. 기능설명서 우선

- GPS, 결제, 안심 채팅, 건강정보, OCR/AI 같은 항목은 디자인보다 기능설명서와 실제 연동 범위가 우선이다.
- 즉, 이 영역은 화면만 먼저 키우는 방식보다 실제 구현 범위를 먼저 정하는 편이 맞다.

## 재정립한 작업 우선순위

### P1. 인증/공통 화면 polish

- 역할 선택
- 권한 안내
- 로그인

목표:
- 불필요한 설명을 줄이고
- 카드 간격과 버튼 우선순위를 단순화
- 현재 구현된 기능 흐름은 유지

### P2. 환자/보호자 홈과 진행 화면 polish

- 홈 상단 상태 카드 강조
- 퀵 액션 카드 2x2 위계 정리
- 예약 확정/진행/실시간 위치 화면의 정보 구획 분리

### P3. 매니저 홈/가이드 polish

- 대기/진행 상태 카드 우선순위 강화
- 액션 카드군 정리
- 가이드 단계 레일과 포커스 카드 시각 위계 보정

### P4. 서류 등록/내 페이지 polish

- 업로드 CTA, 상태 카드, 검토 메모 요약 재배치
- 간호사 자격증/요양보호사 자격증 분기 표현 정리

### P5. 설정/마이페이지 묶음 정리

- 설정, 이용 내역, 정산/문의 묶음 분리
- 카드/섹션 경계를 더 분명하게 정리

## 지금 우선순위에서 밀어도 되는 것

- GPS 지도 SDK 내장 화면
- PG 실결제
- 푸시 알림
- OCR/AI 기능

이 항목들은 디자인보다 실제 연동 범위 결정이 먼저다.

## 결론

- 최신 디자인 세트는 `단일 ZIP 한 장` 기준이 아니라 `분할 화면 세트` 기준으로 봐야 한다.
- 기능 우선순위는 바꾸지 않는다.
- 디자인 기준으로는 다음 작업을 `인증/공통 -> 환자/보호자 홈 -> 매니저 홈/가이드 -> 서류/내 페이지 -> 설정` 순서로 잡는 것이 맞다.
