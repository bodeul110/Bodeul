## 이번 디자인에서 더 분명해진 축

### 1. 인증/공통 화면

- 첫 화면, 로그인, 역할 선택, 권한 안내가 모두 `짧은 설명 + 소수의 핵심 카드 + 분명한 단일 CTA` 구조다.
- 현재 구현은 흐름은 맞지만, 카드 간격과 정보 압축이 더 단정해질 여지가 있다.

### 2. 환자/보호자 홈

- 상단 상태 카드
- 2x2 주요 액션 카드
- 하단 보조 카드

이 3단 위계가 더 강하다. 특히 `환자 홈 화면 (예약 완료 후).png`, `환자 홈 화면 (예약 완료 후)-1.png`가 예약 이후 상태 변형을 따로 보여준다. 현재 구현은 기능은 충분하지만, 상단 상태 카드와 하단 보조 카드의 구획이 더 분명해져야 한다.

### 3. 환자 진행/리포트

- `실시간 진행`
- `병원 편 리포트`
- `복약 변화 리포트`
- `병원 내 지도`

이 네 축이 보드로 분리돼 있다. 현재 구현은 데이터가 더 풍부하지만, 시각적으로는 아직 한 화면 안에 정보가 많이 섞여 있다. `건강리포트- 병원 편`, `건강리포트-복약 변화`, `병원 내 지도`가 각각 독립 보드로 빠져 있으므로, 리포트와 지도는 한 카드에 섞기보다 화면 구획을 더 분리하는 쪽이 맞다.

### 4. 매니저 홈/가이드

- 매니저 홈은 현재 대기/진행 상태를 강하게 보여주는 상단 카드 중심
- 가이드는 단계형 흐름과 지도/메모/사진을 분리해서 보여주는 구획형 구조

현재 구현도 방향은 맞지만, `동행가이드1`~`12`가 사실상 하나의 연속 시퀀스를 이루고 있으므로 단계 타임라인과 지도/메모 블록을 더 분리해 읽기 쉽게 만들 필요가 있다.

### 5. 서류 등록/설정

- 서류 등록은 `파일 종류`, `현재 상태`, `다음 액션` 위계가 선명하다.
- 사용자 설정 보드는 설정을 한 화면에 나열하기보다 묶음 단위로 분리한다.

### 6. 보조 개요 보드

- `Main.png`
- `Body.png`
- `보들 가이드.pdf`

이 세 자산은 개별 화면 설계보다 전체 레이아웃과 카드 간격, 화면군 묶음을 빠르게 비교할 때 더 유용하다. 세부 구현 판단은 개별 PNG를, 전반적인 위계 판단은 이 개요 보드를 우선 보는 쪽이 맞다.

## 유지해야 하는 차이

### 1. 관리자 진입

- 관리자 로그인은 일반 사용자 경로에 노출하지 않는 현재 정책을 유지하는 게 맞다.

### 2. 관리자 웹 분리

- 피그마는 사용자 앱 보드 중심이다.
- 관리자 심사/운영은 계속 `admin-web` 별도 기준으로 두는 편이 맞다.

### 3. 기능설명서 우선

- GPS, 결제, 안심 채팅, 건강정보, OCR/AI 같은 항목은 피그마보다 기능설명서의 실제 연동 범위를 먼저 따른다.

## 정리한 작업 우선순위

### P1. 인증/공통 화면 polish

- 첫 화면
- 로그인
- 역할 선택
- 권한 안내

### P2. 환자/보호자 홈과 진행 화면 polish

- 상단 상태 카드 강화
- 주요 액션 2x2 구획 정리
- 진행/리포트/지도 정보 구획 분리

### P3. 매니저 홈/가이드 polish

- 대기/진행 카드 우선순위 강화
- 단계 타임라인과 지도/메모 카드 분리

### P4. 서류 등록/내 페이지 polish

- 상태 카드, 업로드 CTA, 검토 메모 요약 위계 정리

### P5. 설정/마이페이지 묶음 정리

- 설정, 이용 내역, 정산/문의 묶음 구분 강화
