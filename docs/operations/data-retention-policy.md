# 데이터 보관 및 파기 정책

기준일: 2026-07-18

## 적용 원칙

이 문서는 운영 구현에 적용할 기술 기본값이다. 개인정보 보호법은 보유기간 경과나 처리 목적 달성으로 불필요해진 개인정보를 지체 없이 파기하도록 요구한다. 위치정보는 서비스 제공 목적을 달성한 뒤 원본을 계속 보관하지 않는 것을 기본으로 한다.

출시 전 개인정보 처리방침, 위치기반서비스 이용약관과 실제 구현을 국내 법률 전문가 또는 개인정보 보호책임자가 최종 대조해야 한다. 법령상 별도 보존 의무가 확인되면 해당 자료만 일반 업무 데이터와 분리해 보관한다.

## 권장 보관 기간

| 데이터 | 기본 보관 기간 | 파기 또는 축소 방식 |
| --- | --- | --- |
| 진행 중 최신 위치와 최근 좌표 10건 | 세션 진행 중 | 완료·취소 후 24시간 이내 정밀 좌표 삭제 |
| 위치정보 수집·이용·제공사실 확인자료 | 6개월 | 좌표 원본 없이 주체, 목적, 시각, 제공 대상을 감사 레코드로 보관 후 삭제 |
| 위치 요약 | 세션 리포트 보관 기간 | 정밀 좌표와 전체 이동 경로를 제외하고 병원 도착·이동 완료 같은 사건만 유지 |
| 채팅 본문 | 세션 완료·취소 후 180일 | 만료 시 본문과 검색용 파생값 삭제, 최소 감사 메타데이터만 유지 |
| 채팅 첨부 파일 | 세션 완료·취소 후 30일 | Storage 원본 삭제 후 DB 메타데이터를 `DELETED` 상태로 변경 |
| 매니저 증빙 원본 | 심사 완료 후 30일 | 신분증, 자격증, 건강진단, 범죄경력 원본 삭제 |
| 매니저 심사 결과 메타데이터 | 매니저 자격 유지 기간 + 1년 | 문서 종류, 검증 결과, 검토자, 검토 시각, 유효기간만 유지 후 삭제 |
| 반려·철회된 매니저 신청 | 결정 또는 철회 후 30일 | 원본과 상세 사유 삭제, 최소 감사 이벤트만 1년 유지 |
| 예약·결제·정산 법정 보존 자료 | 적용 법령이 요구하는 기간 | 일반 서비스 DB와 논리적으로 분리하고 목적 외 조회 금지 |
| 운영 로그 | 30일 | 인증 token, 정밀 좌표, 채팅 본문, 서류 경로를 기록하지 않고 순환 삭제 |
| PostgreSQL 외부 logical dump | 4주 | 검증된 최신 4회만 유지하고 순환 삭제 |
| Supabase 제공자 백업 | 7일 | Pro 일일 백업 정책에 따라 만료 |

### 위치

- 현재 앱의 세션당 최근 10건 제한은 유지한다.
- 정밀 좌표를 장기 분석, 추천 또는 성과 평가 데이터로 재사용하지 않는다.
- 세션 완료 또는 취소 이벤트가 발생하면 `expires_at = completed_at + 24 hours`를 기록하고 정리 job이 최신 좌표와 이력 좌표를 삭제한다.
- 위치정보 확인자료는 정밀 좌표가 아니라 수집·이용·제공 여부를 증명하는 최소 감사 데이터로 6개월 보관한다.
- 사고나 분쟁이 접수되면 특정 세션에만 legal hold를 걸 수 있지만, 별도 근거와 종료일을 기록해야 한다.

### 채팅

- 채팅은 별도 PostgreSQL 행으로 저장하고 예약 또는 동행 세션에 연결한다. 배열 필드로 누적하지 않는다.
- 채팅 본문 180일은 CS와 분쟁 확인을 위한 운영 기본값이다. 마케팅, 모델 학습 또는 장기 통계에 사용하지 않는다.
- 첨부는 건강·처방·신분 관련 정보가 포함될 가능성이 있으므로 본문보다 짧은 30일을 적용한다.
- 사용자가 삭제를 요청했거나 계정을 탈퇴한 경우 법정 보존 또는 유효한 legal hold가 없는 데이터는 정기 만료일보다 먼저 삭제한다.

### 매니저 증빙서류

- 승인된 매니저의 원본도 무기한 보관하지 않는다. 심사가 끝나면 30일 안에 원본을 삭제하고 검증 결과와 유효기간만 유지한다.
- 재심사에 원본이 필요하면 새 파일을 다시 제출받는다.
- 주민등록번호 등 고유식별정보가 불필요한 경우 업로드 전에 가리도록 안내하고, 서버에서도 불필요한 OCR 또는 로그 추출을 하지 않는다.
- Storage object 경로를 공개 URL로 만들지 않으며 관리자 서버의 권한 확인 뒤 제한된 시간의 접근 경로만 제공한다.

## 예외 보관

분쟁, 수사 협조, 법적 의무로 기본 기간을 넘겨야 할 때는 다음을 모두 기록한다.

- 대상 사용자와 데이터 범위
- 보관 근거와 담당 승인자
- 시작일과 만료일
- 접근 가능한 운영자
- 해제 및 실제 파기 일시

기술 기본 legal hold는 최대 180일이다. 더 긴 기간은 적용 법령이나 법률 검토 결과를 함께 남겨야 한다. hold 데이터는 일반 조회와 백필 대상에서 제외한다.

## 구현 기준

1. PostgreSQL 테이블에 `expires_at`, `deleted_at`, 필요한 경우 `legal_hold_until`을 둔다.
2. 만료 정리 job은 매일 실행하고 삭제 건수와 실패 건수만 운영 로그에 남긴다.
3. Storage 삭제와 DB 상태 변경은 재시도 가능한 job으로 처리한다.
4. 파기 job은 dry-run 보고서와 실제 apply 모드를 분리한다.
5. 백업 안의 삭제 데이터는 백업 자체 만료 시점까지 존재할 수 있음을 처리방침에 알린다.
6. 월 1회 만료 예정, 삭제 성공, 실패와 legal hold 건수를 운영 보고서로 확인한다.
7. Firestore 전환 기간에도 같은 기간을 적용하며, PostgreSQL 전환을 이유로 만료 데이터를 백필하지 않는다.

## 검증 기준

- 만료 전 데이터는 역할별 정상 조회되고 만료 후에는 조회되지 않는다.
- 다른 사용자의 위치, 채팅과 첨부에 접근하면 403 또는 동등한 거부 응답을 반환한다.
- Storage 삭제 실패가 DB에서 성공으로 표시되지 않는다.
- legal hold가 있는 레코드는 일반 만료 job에서 제외되고, hold 해제 뒤 다음 job에서 삭제된다.
- 파기 결과에는 원문 개인정보 대신 레코드 ID, 정책 코드와 처리 결과만 남는다.
