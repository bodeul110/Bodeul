# 설계 판단 기록 규칙

기준일: 2026-06-25

초기에는 빠른 구현을 우선했기 때문에 모든 선택 근거가 사전에 정리되지는 않았다.
현재는 구현된 구조를 기준으로 선택 이유, 대안, 단점, 전환 조건을 정리하고 있다.

이 문서는 앞으로 기능 개발, 구조 변경, 운영 설정 변경을 할 때 팀이 최소한으로 남겨야 하는 판단 기록 형식이다. 목적은 처음부터 완벽하게 설계한 것처럼 보이게 하는 것이 아니라, 현재 구조를 이해하고 팀의 판단으로 설명할 수 있게 만드는 것이다.

## 기본 원칙

- 기능 개발 전에 “왜 이 작업을 하는지”를 한 줄이라도 쓴다.
- “AI가 제안했다”는 선택 이유가 될 수 없다. AI 제안은 검토 대상이고, 최종 판단은 팀이 책임진다.
- 선택 근거는 항상 현재 팀 규모, MVP 단계, 운영 부담과 연결해서 설명한다.
- 대안과 리스크를 짧게라도 남긴다. 당장 선택하지 않은 이유와 나중에 바꿀 조건이 중요하다.

## PR과 작업 문서 기준

일반 PR은 아래처럼 짧게 쓴다. 작업과 무관한 빈 구획은 삭제한다.

```markdown
## 배경

무엇을 해결하려는 작업인가?

## 변경 내용

무엇이 달라졌는가?

## 확인

실제로 무엇을 실행하거나 확인했는가?

## 참고할 점

리뷰가 필요한 판단, 리스크, 후속 작업이 있는가?
```

설계, 보안, 인프라처럼 선택 자체가 중요한 작업은 본문이나 별도 문서에 선택한 방식, 대안, 선택 이유, 리스크를 덧붙인다. 모든 PR에 같은 장문 형식을 반복하지 않는다.

## 좋은 설명 기준

나쁜 설명:

> AI가 이렇게 하라고 해서 했습니다.

좋은 설명:

> AI가 제안한 구조를 검토했고, Activity에 로직이 몰리는 문제를 줄이기 위해 Coordinator와 Repository로 역할을 나누는 방식을 유지하기로 했다. 다만 파일 수가 많아지는 단점이 있어 화면별 구조 문서를 추가했다.

## 현재 규모와 연결하는 표현

- 현재 MVP에서는 Firebase Auth와 FCM을 유지해 인증과 알림 전환 위험을 줄인다.
- 운영 DB는 관계형 조회와 감사 이력을 위해 Supabase PostgreSQL로 단계 전환한다.
- 관리자 서버와 Spring Core API를 분리하되 서버가 다른 서버를 거쳐 DB로 가는 중복 경로는 만들지 않는다.
- 현재 매칭은 추천 모델이 아니라 규칙 기반 운영 로직으로 보고, 데이터가 충분히 쌓인 뒤 추천 모델을 검토한다.

## 문서 위치 기준

- 인프라/런타임 판단: `docs/architecture/`
- 운영 절차와 검증: `docs/operations/`
- 실제 실행 결과: `docs/reports/`
- 보안 경계와 리스크: `docs/security/`
